Context

Why ownership concentration matters

ONBOARDING

When a critical module is effectively owned by one person, new contributors can't ramp up on it without that person's involvement. Reviews bottleneck. Knowledge doesn't transfer.

CONTRIBUTOR DEPARTURE

When that contributor leaves — changes job, loses interest, or becomes unavailable — maintenance of that module effectively stops or shifts entirely to whoever is willing to learn it from scratch.

SCALE DOESN'T HELP

The assumption is that large, popular projects are safe from this. This dataset suggests otherwise. Having thousands of contributors does not appear to prevent ownership concentration in individual modules.

Key Findings

What the data shows

Four findings emerge cleanly from the dataset. None of them require extrapolation — they are direct observations from the numbers.

01
The "more contributors" assumption doesn't hold

Every single project — from Rails with 6,483 contributors to Remix with 29 — had at least one module where a single author owns more than 50% of the change history. Contributor count and ownership concentration are largely independent variables in this dataset.

02
Spring Boot showed the highest BF-1 module count in this dataset

25 bus factor 1 modules with only 140 authors — the worst ratio in the entire dataset. This number will surprise readers. It held up across three separate analysis runs. Spring Boot's architecture is highly modular: 22,528 files spread across dozens of sub-modules, many of which appear to have been authored and maintained by a small number of contributors. The raw data is in research/spring-boot.json.

03
Vite is the positive outlier

236 authors, only 1 bus factor 1 module. The healthiest large project in the dataset by a significant margin. This is what deliberate code ownership looks like — and it shows the pattern can be broken.

04
Scale provides no immunity

VSCode (3,050 contributors → 15 BF-1 modules), Kubernetes (5,358 → 9), Rails (6,483 → 7). No clear relationship between contributor count and BF-1 module count was observed in this dataset.

Methodology

How this was measured

All analysis was performed using git-archaeologist v1.9.2 (git-arch analyze <repo> --json) against local clones of each repository. Repository commit SHAs at time of analysis are recorded in the raw data.

Bus factor definition

A module is classified as bus factor 1 when a single contributor accounts for more than 50% of all non-merge commits touching that file. This is a conservative threshold — it measures observable change authorship, not theoretical knowledge.

Curse score formula

The curse score is a composite metric computed by git-archaeologist for each file:

curseScore = totalChanges × log2(authors+1) × exp(-0.5 × ageInYears) × log2(churn+2) × acceleration × 100

where:
  totalChanges  = count of non-merge commits touching the file
  authors       = number of unique commit authors
  ageInYears    = time since file's first commit (exponential decay — old stable files score lower)
  churn         = change frequency derived from commit timeline
  acceleration  = whether the file is getting more dangerous recently vs historically

Source: src/analyzers/curseScorer.ts

The exponential decay on age means old chaos that has since stabilized does not dominate the ranking. The acceleration multiplier means files getting worse recently score higher than files with similar totals that have levelled off. This matches the formula documented in the README exactly.

What counts as a commit?
All non-merge commits. Merge commits are excluded to avoid double-counting changes introduced in feature branches.
Are bots excluded?
Yes. Commits from accounts matching bot patterns (dependabot, renovate, github-actions, and similar) are excluded from authorship calculations.

What files are excluded from cursed rankings?
Generated files, config files, documentation, lock files, vendored code, and build scripts are filtered. Only source files are ranked.
Is this causation or correlation?
Correlation. This data shows that ownership concentration co-exists with large contributor bases — it does not explain why, or assert that concentration causes failures.
What was excluded entirely?
Rust (168k files — exceeded memory limits). All other repos were analyzed successfully.

Limitations

What this data does and doesn't show

Publishing limitations is not a hedge. It is the difference between a credible analysis and a marketing document. These are the honest constraints on what the numbers mean.

Analysis window varies by repo
git-archaeologist analyzed commits up to its internal limit per run. Several repos show round commit counts (1,000; 2,000) — these reflect the tool's analysis window, not the total repository history. React has ~20,000 real commits; TensorFlow has ~170,000. The BF-1 calculations are based on the analyzed window, not lifetime history. This is the most important limitation in this dataset. Full history re-runs are planned for v2.0.
Authorship ≠ knowledge
A bus factor 1 classification means one author owns the majority of recorded commits to a file. It does not measure who has reviewed that code, who has read it, or who could maintain it. Commit authorship is an imperfect proxy for knowledge — but it is an objective, reproducible one.
Repositories are a snapshot in time
All 26 repos were analyzed at specific commit SHAs on June 9, 2026 (recorded in research/README.md). Results will differ if you re-run after significant contributor activity. Ownership concentration can improve — Vite proves this.
Authors are identified by commit signature
The same person contributing under two different email addresses counts as two authors. Co-authored commits (common in some projects) are attributed to the first listed author. Both factors could slightly undercount real unique contributors.
Files analyzed are source files only
Generated files, vendored code, config, docs, and lock files are excluded from the cursed file rankings and from BF-1 counts where possible. The filter is a best-effort heuristic — some edge cases may have slipped through. The full filter pattern list is in the Methodology section.
Correlation, not causation
This data shows that bus factor 1 modules co-exist with large contributor bases across every project analyzed. It does not explain why this occurs, nor does it predict failures. A bus factor 1 module is a risk indicator, not a guarantee of problems.

On round commit numbers: repos showing exactly 1,000 or 2,000 commits were analyzed within git-archaeologist's default window. The full repository has more history. The bus factor and curse score findings for those repos are valid within the analyzed window — but the commits and authors figures in the table reflect the window, not the total project lifetime. A footnote marker (†) appears next to those rows in the data table.

Reproducibility

Raw data & reproduction

Per-repository JSON outputs from git-archaeologist are published alongside this report. Each file contains the full analysis: cursed files array, bus factor breakdown, file statistics, and the commit SHA of the repository at time of analysis.

npm install -g git-archaeologist
git-arch analyze /path/to/repo --json

Repository commit SHAs at time of analysis are recorded in research/README.md in the project repository. All 26 JSON files are available individually.
